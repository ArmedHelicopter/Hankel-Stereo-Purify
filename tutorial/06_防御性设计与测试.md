# 第 6 章：防御性设计与测试

## 直觉先行

外部文件路径、声道数、格式、内存上限都是**不可信输入**。系统用**分层异常**（配置错 vs I/O 错 vs 数值错）告诉调用者「哪类失败」，而不是笼统 `Exception`。测试不写「能跑就行」，而是对黄金模型、边界条件、猴子补丁生产者失败路径做**可重复**断言。

## 决策复盘（Why vs Why not）

**为什么要 `HankelPurifyError` 子类树？**  
CLI 能映射退出码（配置 2、其它 1），日志能分类打点。吞异常违反本仓库工程底线；门面 [`process_file`](../src/facade/purifier.py) 把 numpy/scipy 数值失败归一成 `ProcessingError`，避免调用方解析字符串。

**为什么 stereo 强制 2 声道？**  
MSSA 联合块假设左右成对；单声道/surround 在 [`require_stereo_channels`](../src/io/stereo_soundfile.py) 处**早失败**，比在 SVD 里崩更省算力。

**为什么测试里大量 `tmp_path` + 小 FLAC？**  
集成测试要**快、可 CI**；小矩阵 golden 检验放在 `tests/test_*`，与 PRD 大文件验收分工明确。

### 代码锚点

| 主题 | 定位 | 路径 |
|------|------|------|
| 异常分层与基类 | `HankelPurifyError` 子类树 | [`src/core/exceptions.py`](../src/core/exceptions.py) |
| CLI 退出码 | `ConfigurationError` → 2，其它 `HankelPurifyError` → 1 | [`src/cli.py`](../src/cli.py) |
| I/O 白名单、扩展名 | `validate_io_paths` | [`src/io/audio_formats.py`](../src/io/audio_formats.py) |

### 测试命题思路（各一例）

- **`tests/test_purifier_producer_errors.py`**：用 `monkeypatch` 或故障注入让生产者线程在解码/入队路径失败，断言门面**不吞异常**、消费者能随毒丸或错误路径退出——验证并发协议与错误传播，而非「 happy path 能跑完」。
- **`tests/test_cli_unit.py`**：对 `main` 或解析层传入非法参数/缺失文件，断言 **退出码** 与异常类型映射与 [`src/cli.py`](../src/cli.py) 合同一致——把 CLI 当 API 测。

- **晦涩**：[`linalg_errors.py`](../src/core/linalg_errors.py) 对 SciPy 可选类型 try/import，新手可能忽略「环境差异」。  
- **建议**：读一遍并理解「为何 ARPACK 错误也要进集合」。

## 收束：如何自己复刻一条最小链

1. 合成 `(F,2)` 白噪声，`AHankelStage(L)` → `BMultichannelStage` → `CSVDStage(FixedRankStrategy(k))` → `DDiagonalStage`。  
2. 对照 [`tests/test_pipeline_mssa.py`](../tests/test_pipeline_mssa.py) 与 [`scripts/benchmark_pipeline.py`](../scripts/benchmark_pipeline.py)。  
3. 再打开 CLI，追 `src/cli.py` → `MSSAPurifierBuilder` → `process_file`。

至此，**波形 → 矩阵 → 子空间 → 波形** 的闭环与工程外壳已对齐。
