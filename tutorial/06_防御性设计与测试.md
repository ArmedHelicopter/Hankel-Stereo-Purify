# 第 6 章：防御性设计与测试

## 直觉先行

外部文件路径、声道数、格式、内存上限都是**不可信输入**。系统用**分层异常**（配置错 vs I/O 错 vs 数值错）告诉调用者「哪类失败」，而不是笼统 `Exception`。同一类门面错误（例如都叫「处理失败」）若只看人读字符串，无法区分根因是 `MemoryError` 还是某依赖抛的 `KeyError`——因此在包装为 `ProcessingError` 时，额外带上**机读的原始异常类型名**（`module.QualName`），与**给人看的文案**解耦。

测试不写「能跑就行」，而是对黄金模型、边界条件、猴子补丁生产者失败路径做**可重复**断言。

## 决策复盘（Why vs Why not）

**为什么要 `HankelPurifyError` 子类树？**  
CLI 能映射退出码（配置 2、其它 1），日志能分类打点。吞异常违反本仓库工程底线；门面 [`process_file`](../src/facade/purifier.py) 把 numpy/scipy 数值失败归一成 `ProcessingError`，避免调用方解析**消息字符串**。

**为什么 `ProcessingError` 还要有 `origin_exception_type`？**  
`ProcessingFailureCode`（如 `UNEXPECTED`）是**粗分桶**，适合指标与日志检索；`str(ProcessingError)` 仍保持简短英文句，不破坏对外习惯。`origin_exception_type` 存 **`exception_fully_qualified_name(exc)`**（见 [`exceptions.py`](../src/core/exceptions.py)），例如 `builtins.MemoryError`、`numpy.linalg.LinAlgError`，便于脚本与运维按**类型**过滤，而不依赖文案或 `__cause__.__class__` 的逐层判断。

**为什么 CLI 先打 `Origin exception type` 再打 `Caused by [...]`？**  
先给出**稳定类型标签**（一行），再给**栈顶定位**与异常 repr——见 [`src/cli.py`](../src/cli.py) `except HankelPurifyError` 分支顺序。

**为什么 stereo 强制 2 声道？**  
MSSA 联合块假设左右成对；单声道/surround 在 [`require_stereo_channels`](../src/io/stereo_soundfile.py) 处**早失败**，比在 SVD 里崩更省算力。

**为什么测试里大量 `tmp_path` + 小 FLAC？**  
集成测试要**快、可 CI**；小矩阵 golden 检验放在 `tests/test_*`，与 PRD 大文件验收分工明确。

### 测试命题思路（各一例）

- **`tests/test_purifier_producer_errors.py`**：用 `monkeypatch` 注入 `_run_processing` 失败，断言 `ProcessingError.origin_exception_type` 与 `numpy.linalg.LinAlgError` / `RuntimeError` 等一致——**机读类型**与 `match="Unexpected error"` 人读文案可同时满足。
- **`tests/test_exceptions.py`**：`exception_fully_qualified_name` 与 `ProcessingError` 可选字段的最小契约。
- **`tests/test_process_frame.py`**：单帧链与 `AudioPurifier._make_denoise_frame_fn` 可运行性。
- **`tests/test_cli_unit.py`**：对 `main` 或解析层传入非法参数/缺失文件，断言 **退出码** 与异常类型映射与 [`src/cli.py`](../src/cli.py) 合同一致——把 CLI 当 API 测。

## 思维挂钩

| 代码 | 专业课 | 软件工程 |

| `MSSA_LINALG_ERRORS` / `MSSA_ARPACK_ERRORS` / `MSSA_NUMERIC_STEP_ERRORS` | dense LinAlg + 稀疏 `svds` ARPACK | 门面显式分支映射；扩展时在 `linalg_errors` 增类型 |

| `monkeypatch` 生产者 | — | 隔离 I/O、测并发协议 |
| `pytest` + coverage | — | 回归即文档；重构安全网 |

## 晦涩点与建议

- **晦涩**：[`linalg_errors.py`](../src/core/linalg_errors.py) 将 NumPy/SciPy `LinAlgError` 与 `ArpackError` / `ArpackNoConvergence` 分 tuple 聚合，门面分别记录日志。  
- **建议**：读一遍并理解「为何 ARPACK 错误也要进集合」。
- **晦涩（能量路径）**：`c_svd` 中 `_energy_truncated_factors` 的 `k_probe` / `margin` 与 `_SVDS_ENERGY_PROBE_CAP`——数学上仍按能量规则选秩，实现上用**常量帽**限制最坏情况下的反复 `svds`。  
- **建议**：用 [`scripts/benchmark_pipeline.py`](../scripts/benchmark_pipeline.py) 在目标 `L,K` 上观察阶段 C 墙时；不必在教材中手推闭式，但要建立「启发式 + 容差」意识。

## 收束：如何自己复刻一条最小链

1. 合成 `(F,2)` 白噪声，顺序调用 `hankel_embed` → `combine_hankel_blocks` → `make_svd_step(..., FixedRankStrategy(k))` 闭包 → `diagonal_reconstruct`，或直接调用 [`process_frame`](../src/core/process_frame.py)。  
2. 对照 [`tests/test_pipeline_mssa.py`](../tests/test_pipeline_mssa.py)、[`tests/test_process_frame.py`](../tests/test_process_frame.py) 与 [`scripts/benchmark_pipeline.py`](../scripts/benchmark_pipeline.py)。  
3. 再打开 CLI，追 `src/cli.py` → `AudioPurifier` → `process_file`。

至此，**波形 → 矩阵 → 子空间 → 波形** 的闭环与工程外壳已对齐。
